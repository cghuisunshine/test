Project Overview

The service is a split full-stack TypeScript application. The frontend is a role-aware single-page app. The backend exposes authenticated route groups for admin and client features, validates Supabase bearer tokens, and reads or writes operational data through Prisma against PostgreSQL.

  • Frontend
    React + Vite SPA for admin and client experiences.
  • Backend
    Hono API with shared, admin, and client route namespaces.
  • Authentication
    Supabase session handling in the browser and token validation in the API.
  • Persistence
    Prisma client and PostgreSQL for the core data model.

Frontend responsibility

Render the client and admin portal, manage role-based routing, load the active user session, and call backend endpoints with a bearer token through shared hooks and API helpers.

Backend responsibility

Enforce auth and admin checks, expose route groups, orchestrate controller and service logic, and persist records, surveys, documents, notifications, and timelines.

For Users

Guide for All User-Facing Roles

In this project, documentation for clients should be read broadly as documentation for all user-facing roles. The code currently supports four roles: admin, inspector, assistant, and client. The first three are routed into the admin workspace, while client users are routed into the protected client portal.

Role map

  • Admin: full operational access to the admin workspace.
  • Inspector: treated as an authorized admin-side role by protected routing.
  • Assistant: treated as an authorized admin-side role by protected routing.
  • Client: uses the client portal to provide project information and materials.

Workspace split

  • The admin workspace includes dashboards, users, strata, appointments, documents, questions, timelines, and profile management.
  • The client portal focuses on project submission and review workflows such as documents, surveys, timelines, and inspection scheduling.
  • Both surfaces rely on the same Supabase-authenticated session and backend API.

What client users use the portal for

  • Review overall progress from the Dashboard.
  • Confirm strata details and strata members.
  • Upload requested supporting documents.
  • Complete survey sections tied to their project.
  • Provide important timeline dates.
  • Review or respond to inspection and appointment steps.

Known status note

The current client routes include a Reports page, but the UI labels it as Coming Soon. Documentation should not describe reporting as a completed client feature.

Main Role-Based Workflows

These workflows are based on the existing route definitions and current page behavior in the frontend codebase. Admin-side roles and client users do not see the same surface area, so the documentation should distinguish between the admin workspace and the client portal.

Client portal workflows

Dashboard /client/dashboard See project progress, action items, document status, survey completion, and appointment-related alerts.

Client access flow

Authentication starts through public routes like /login, /accept-invite, /set-password, and /auth/callback. Once the browser has a valid Supabase session, the app resolves the user profile and routes the user to the correct experience.

For Future Developers

Developer Handoff Guide

This project already has a clear split between frontend and backend. The main maintenance task is keeping the route contracts, auth assumptions, and shared domain behavior aligned across both sides.

Frontend structure

frontend/src is divided into admin/, client/, and shared/. Shared code holds auth context, reusable components, hooks, route guards, API helpers, and global SCSS.

Backend structure

backend/src is divided into admin/, client/, and shared/. Route files delegate to controllers, and controllers rely on shared services, helpers, middleware, and types.

Runtime topology

[Browser SPA]
  -> Supabase Auth
  -> Backend API with bearer token
     -> authMiddleware
     -> adminMiddleware for /admin/*
     -> Controllers
     -> Shared services
     -> Prisma client
     -> PostgreSQL

Routing and auth model

  • Public frontend routes include login, invite acceptance, password setup, password reset, privacy, terms, and auth callback handling.
  • Client frontend routes are protected with ProtectedRoute and wrapped in ClientLayout.
  • The browser auth context reads the Supabase session, then loads role-specific profile data.
  • The backend validates bearer tokens with Supabase and applies an additional admin role check for /admin/*.
  • Shared backend routes currently include lookups and notifications, while admin and client routes expose the role-specific features.

Route namespaces to know

Frontend

  • /login, /set-password, /reset-password, /auth/callback
  • /dashboard as the role-aware landing route
  • /admin/* for administrator tools
  • /client/* for the client portal

Backend

  • /health for health checks
  • /api/lookups/* for shared lookup data
  • /admin/* for admin-only endpoints
  • /client/* for authenticated client operations

Developer Setup and Local Commands

The frontend and backend are separate Node projects. Install dependencies and run each side from its own directory.

Frontend

cd frontend
npm install
npm run dev
npm run build

Backend

cd backend
npm install
npm run dev
npm run build
npm run start

Frontend VITE_SUPABASE_ANON_KEY Supabase public anonymous key used by the browser client.
Backend PORT Optional API port. Defaults to 3000.

Maintenance Hotspots

These are the parts of the codebase where changes are most likely to ripple across the system and require careful verification.

Auth and routing

Changes to auth, role detection, or redirect behavior affect public routes, protected route guards, the auth context, and backend middleware together. Keep frontend session assumptions aligned with backend token validation.

Documents and surveys

The client dashboard, documents flow, and survey flow all reflect status from the same active request. Changes here often require matching updates in frontend hooks, backend route handlers, and service logic.

Lookup-driven UI

Survey sections and other UI labels depend on backend-provided lookup data. If lookup shapes change, verify the user-facing section labels and completion logic still behave correctly.

Prisma-backed behavior

Schema or relation changes can affect middleware role checks, profile loading, service methods, and any route returning nested project data. Review migrations, types, and include selections together.

Recommended change checklist

  • Confirm the route exists on both sides when changing a frontend workflow that calls the API.
  • Check role restrictions before exposing or moving any route.
  • Verify environment variables and local defaults when changing auth, CORS, or database access.
  • Keep documentation aligned with the code, especially when a feature is still partial or marked as coming soon.
